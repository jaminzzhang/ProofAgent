1. 已确认的发布范围

身份与权限

S2 待实现
  • OIDC 独占认证，登录后 7 天有效。
  • 只做权限映射，不做本地账号、密码、用户目录或用户管理页。
  • Dashboard 直接配置权限，无审批工作流。

产品表面

S0 已收口
  • 保留 Dashboard 与 `/operator`。
  • 删除 Customer Chat、handoff、审批队列和 approve/deny。
  • 删除旧示例和 V1/V2 workflow。

执行与响应

S4 待实现
  • Run Executor 是同镜像进程角色，不拆独立微服务。
  • Proof Agent 内建 PostgreSQL 持久异步队列。
  • 请求立即返回，SSE 推送粗粒度进度。

状态与制品

S1 + S3
  • PostgreSQL 管可变事务状态。
  • S3 管不可变 trace、receipt、knowledge 等制品。
  • S3 写入并验证后再更新 PG；允许丢弃未提交的部分进度。

工具边界

S0 基线
  • 初版不允许本地 Python handler、stdio 或状态变更工具。
  • 未来只可加入经发布验证的只读 HTTPS/MCP 工具。
  • 不建设工具审批系统。

未来沙箱

不在初版
  • 脚本/命令执行必须独立威胁模型与隔离边界。
  • 需要 CPU/内存/时长/磁盘限额、默认拒绝网络和全量审计。
  • 不能复用 Agent/API/Worker 进程直接执行。

5. 不可打乱的前后依赖

S0 已完成唯一 V3 基线与 Gate 合同
S1PG store ports / migrations
S2 ∥ S3身份安全与 S3 制品可并行
S4队列 / Executor / SSE
S5唯一生产 Agent
S6部署恢复与正式 Gate

为什么这样排

  • S2 的 session/permission 和 S3 的 artifact metadata 都需要 S1 的 PostgreSQL 事务与迁移基础。
  • S4 必须同时冻结权限/secret/egress 语义和 artifact commit 语义，因此依赖 S2 + S3。
  • S5 需要真实队列执行和 S3 knowledge/materialization，不能提前把本地 package 当生产 Agent。
  • S6 才能绑定最终镜像、外部依赖组合、迁移、Agent bundle 和 Gate 证据。

S2 · OIDC、权限、Secrets、Egress

  2. CSRF、cookie flags、session revocation/freshness、logout。

S3 · S3 制品、保留与恢复

  1. Artifact Port 支持 immutable key、version、length、digest。
  2. S3-first object + manifest verification 后 PG 可见性事务。
  3. verified read-only materialization cache，损坏/缺失 fail closed。
  4. orphan GC、retention TTL、legal/audit hold、exact-version restore。
  5. PG PITR 与 S3 版本联合恢复，目标 RPO≤15m、RTO≤4h。
验收：真实 S3-compatible 集成、故障、GC、TTL 和 restore 测试。

S4 · Async Queue、Executor、SSE

  5. coarse durable state + best-effort detail SSE；断线重连。

7. 当前风险与收口债务

P0 · 生产基础设施尚未实现

当前 local stores、package knowledge 和 deterministic provider 只能证明开发基线，不能承担生产事务、身份、制品、恢复或容量语义。

P1 · Dormant vocabulary 与内部兼容痕迹

部分历史 contracts、stage descriptors、ADR/spec 仍含 approval/customer/LangGraph 词汇。历史文件可保留；活跃代码若仍暴露不可达 approval 状态，应在 S1/S2 前继续缩窄并用 inventory guard 固化。

P1 · 验证噪声

必须在可监听 socket 的 CI/host 跑完 8 项；修复 Pydantic serializer warning 和 React act(...) warning，决定 Chat code-splitting 预算。

P1 · 文档与 CLI 漂移

本次已重写活跃事实入口并修正 demo 帮助文本；后续行为变更必须同提交更新文档，避免规划再次冒充实现。

P2 · 未来沙箱复杂度

不要把沙箱夹带进 S1–S6。先完成初版发布，再单独做 threat model、isolation PoC、escape tests、resource abuse tests 与 artifact exchange design。

8. 发布前最终验收清单

  • □ clean source commit、clean tree、版本和 OCI digest 完整绑定。
  • □ wheel、Dashboard、Operator Chat、migrations、Agent bundle、evaluation contract 均有 digest。
  • □ 生产启动不能回退到 local identity/filesystem secrets/package knowledge/local tools。
  • □ OIDC 登录、7 天 session、logout/revoke、CSRF、permission negative、recovery group 演练通过。
  • □ PG/S3/OIDC/secret/gateway/model 具体版本组合写入 compatibility manifest。
  • □ 20 online / 5 active / 50 queued，超限明确拒绝。
  • □ SSE 首进度、重连、取消、deadline、worker crash/fencing 全部通过。
  • □ 每个成功或 governed terminal result 都有 verified artifact manifest；否则 result_available=false。
  • □ PITR + exact S3 version 联合恢复满足 RPO/RTO。
  • □ Blue/Green standby、drain、switch、soak、rollback 完整演练。
  • □ Customer 和 Approval 路由在生产仍不可达。
  • □ 3–5 名内部 operator 完成一个支持日 pilot。
  • □ 13 个 Gate 全部 passed、fresh、阈值满足且绑定同一 candidate。
  • □ proof-agent release verify 输出 GO。
