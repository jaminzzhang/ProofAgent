## Problem Statement

Build an open-source **Enterprise Agent Delivery Kit** for enterprise AI Agent teams. The first release should feel like a runnable delivery artifact, not a generic governance SDK or control-plane slide deck.

The first proof point is an enterprise knowledge Q&A Agent that is directly usable after configuring orchestration flow, Harness rules, knowledge base, session memory, one MCP mock tool, trace output, and a human-readable Governance Receipt.

The kit is powered by a **Control Envelope**: it wraps workflow, knowledge, memory, tools, and model output with policy, evidence, approval, and audit.

The Control Envelope is implemented through **Harness Engineering** — the design discipline of inserting typed policy decision points into the Agent workflow. At each critical step (retrieval, answer generation, tool calls, memory writes), the Harness evaluates enterprise rules and produces a typed decision (`allow`, `deny`, `require_approval`, `escalate`). This makes the Agent flow explicitly controlled, not just orchestrated.

**Harness RAG** is the first application of this approach: an **Agentic RAG** implementation where the retrieval-generation pipeline is governed by Harness policy gates. Unlike Plain RAG (retrieve → generate with no control), Harness RAG enforces mandatory retrieval, evidence quality checks, citation requirements, and refusal on weak evidence — ensuring knowledge is controlled, not just retrieved.

The target user is an enterprise AI Agent owner or AI application platform owner who wants to build strongly controlled AI Agents for scenarios such as insurance digital workers, customer service, and internal knowledge assistants.

## What Makes This Cool

The "whoa" moment is not that the Agent answers questions. Ordinary RAG demos already do that. The moment is that every answer is governed and explainable:

- The workflow is explicit and inspectable.
- Knowledge retrieval can be mandatory before answer generation.
- Answers must include citations when policy requires them.
- Tool calls can require permission or human confirmation.
- Memory writes are bounded by rule.
- Trace/audit output shows what happened before, during, and after generation.
- A Governance Receipt summarizes the run for AI Agent owners and architects.

## Constraints

- First release must feel open-source and runnable, not like a slideware architecture.
- The framework should reuse existing ecosystem pieces instead of rebuilding them: LangGraph for workflow/runtime, LlamaIndex or equivalent for knowledge/RAG, MCP for tools, and Langfuse/OpenTelemetry-style adapters for traces.
- The first demo is enterprise knowledge Q&A because it is broadly understandable across insurance, finance, manufacturing, and government-enterprise contexts.
- The project must avoid looking like a generic RAG template or another low-code Agent builder.

## Premises

1. The first user is the enterprise AI Agent/platform owner, not a generic hobby Agent developer.
2. The first demo is enterprise knowledge Q&A, but it must be Harness-controlled RAG, not plain RAG.
3. The MVP differentiation is rule execution evidence: retrieval, policy checks, memory boundaries, tool control, citation, and trace.
4. The framework value sits above mature components as a policy, orchestration, and audit envelope.
5. Distribution must be designed from day one: GitHub repo, Docker Compose, one-command demo, and minimal Python CLI.

## Landscape Notes

LangGraph already covers low-level orchestration primitives such as durable execution, persistence, and human-in-the-loop. Dify covers visual workflows, RAG, tools, and observability. LlamaIndex covers RAG, agents, and workflows. Langfuse covers tracing, evaluation, and LLM observability. MCP standardizes how LLM applications reach tools and resources.

The gap is not another component. The gap is a default enterprise execution envelope that composes these pieces into a governed delivery artifact.

## Approaches Considered

### Approach A: Governance-First Demo Kit

Start with one strong enterprise knowledge Q&A demo. Build only the framework abstractions needed to support that demo.

Pros: fastest proof, low risk, easy README and video.  
Cons: initial framework may be narrow and require later refactoring.

### Approach B: Pluggable Harness Framework

Design the framework first: runtime adapter, policy engine, memory interface, knowledge interface, MCP gateway, and audit interface. Ship the enterprise knowledge Q&A Agent as the first template.

Pros: stronger long-term open-source shape, clearer extension model, better fit for multiple enterprise templates.  
Cons: higher risk of architecture-first work unless the first demo remains a hard acceptance test.

### Approach C: Policy Playground + Demo

Build an interactive policy playground that lets users change Harness rules and immediately see Agent behavior change.

Pros: strongest demonstration of rule-driven Agent behavior.  
Cons: UI scope can distract from the framework and make the MVP feel like a toy if policy semantics are thin.

## Recommended Approach

Use a **narrowed Enterprise Agent Delivery Kit MVP**, with one non-negotiable constraint: the kit is not accepted until the enterprise knowledge Q&A template runs end-to-end and the user can inspect why it answered, refused, or waited for approval.

The right shape is delivery-kit-first, demo-verified. The open-source repo should expose only the extension points proven by the first template:

- LangGraph-backed workflow execution as the only public v1 runtime.
- `PolicyEngine`: typed decisions for retrieval, answering, tools, memory, and escalation.
- `KnowledgeProvider`: local docs and vector search first, external providers later.
- `MemoryProvider`: session memory first, user/task memory later.
- `ToolApproval`: MCP mock tool registry with approval as an explicit workflow state.
- `AuditSink`: local JSONL trace schema first, Langfuse/OpenTelemetry adapters later.
- `Template`: enterprise knowledge Q&A as the reference implementation.
- `GovernanceReceipt`: human-readable summary generated from trace events.

## CEO Review Decisions

Review mode: **SELECTIVE EXPANSION**. The baseline remains the narrowed delivery-kit MVP, but the public narrative changes from generic governance/control-plane language to Enterprise Agent Delivery Kit.

Accepted expansions:

- **Governance Receipt Contract:** every run generates a human-readable receipt derived from trace events, with required sections, outcome enum, redaction behavior, and tests.
- **Plain RAG vs Harness RAG demo:** the enterprise Q&A demo must show the behavior difference between ordinary RAG and Harness-controlled RAG.
- **3-minute launch path:** README and demo docs must show commands, demo questions, expected answer/refusal/approval behavior, receipt path, and smoke tests.
- **Trust Boundaries:** v1 must document what the kit controls, what it records, and what it does not claim to solve.

Deferred expansions:

- multiple policy packs
- public multi-runtime support
- multiple production providers
- GUI policy playground
- full production MCP Gateway
- 3-5 enterprise templates

## Engineering Review Decisions

- Policy decisions are typed and traceable: `allow`, `deny`, `require_approval`, and `escalate`, each with a reason and trace fields.
- Policy enforcement points exist at `before_retrieval`, `before_answer`, `before_tool_call`, and `before_memory_write`.
- LangGraph is the only public v1 runtime; runtime adapter boundaries stay internal until another runtime is justified.
- MCP approval is an explicit workflow state with requested, granted, denied, and timed-out trace events.
- Evidence quality is part of the v1 contract: weak or missing evidence must cause refusal or escalation, not unsupported answers with decorative citations.
- JSONL trace is the source of truth for v1 audit. External observability tools are adapters, not core dependencies.
- The enterprise QA template uses one top-level `agent.yaml` manifest that references policy, knowledge, tools, and trace settings.
- The first public story is "Enterprise Agent Delivery Kit": run a governed enterprise Agent in 30 minutes, then inspect why it answered, refused, or waited for approval.
- v1 includes a Plain RAG vs Harness RAG comparison to make the Control Envelope difference visible.
- v1 includes a 3-minute launch script and a Trust Boundaries page.

## Success Criteria

- A new developer can run the demo in 30 minutes or less.
- The demo answers enterprise knowledge questions only after following the configured Harness rules.
- Every answer shows citations or a policy-based refusal.
- The trace clearly records workflow nodes, retrieval results, policy decisions, memory reads/writes, tool calls, and final output.
- At least one MCP mock tool demonstrates controlled tool invocation.
- Every run produces a Governance Receipt.
- The demo includes a Plain RAG vs Harness RAG comparison.
- The README includes a 3-minute launch path with expected answer, refusal, approval, trace, and receipt outputs.
- Governance Receipt has a minimal contract that maps receipt sections to trace events.
- Trust Boundaries clearly distinguish v1 controls from deferred security/platform capabilities.
- The README explains why this is different from plain RAG, LangGraph-only orchestration, and Dify-style app building.
- First-run and runtime performance stay within the engineering budgets below.

## Distribution Plan

Ship as a GitHub open-source repository with:

- `docker compose up` for the full local demo.
- `proof-agent run examples/enterprise_qa/agent.yaml` for CLI execution.
- Python package structure for framework code.
- `examples/enterprise_qa/` containing sample docs, policy file, flow config, MCP mock server, expected trace output, expected Governance Receipt, and side-by-side comparison scenarios.
- GitHub Actions for lint, tests, and package build.
- Release artifacts on GitHub Releases once the CLI and Docker demo stabilize.

## Next Steps

1. Define the smallest public framework contract: `agent.yaml`, policy decisions, local knowledge, session memory, MCP mock tools, JSONL audit, Governance Receipt, and template manifest.
2. Write the enterprise Q&A policy examples before implementation: mandatory retrieval, citation required, refusal on missing evidence, approval before tool calls, bounded memory writes.
3. Design the JSONL trace schema, Governance Receipt, and sample trace output early; this is the main proof of control.
4. Define the 3-minute launch script and README smoke test before implementation starts.
5. Define Trust Boundaries before adding tool or memory behavior.
6. Implement the demo path first against LangGraph, local files, local vector storage, session memory, and MCP mock tools.
7. Add Plain RAG vs Harness RAG comparison scenarios to the enterprise Q&A demo.
8. Only add provider plugins or public runtime adapters after the demo proves the interface is real.

## Open Questions

- Should MCP approvals be driven by CLI prompts only in v1, or should there also be a minimal approval API?
- Which evidence thresholds are deterministic enough for tests versus LLM evals?
- What is the minimum Governance Receipt format that satisfies AI Agent owners without becoming a report generator?
- Which external providers should be prioritized after local v1: Langfuse, OpenTelemetry, hosted vector DB, or external memory stores?
