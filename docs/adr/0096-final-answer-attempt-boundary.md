# Final Answer Attempt Boundary

Controlled ReAct V3 will model final-answer generation as a governed Final Answer Attempt before committing the terminal run outcome. The attempt owns model-call permission, Harness-normalized output, validation status, optional repair eligibility, admission status, sensitive LLM interaction capture, and trace-safe failure diagnostics; `ReceiptOutcome` remains the final terminal outcome after governed attempts are exhausted or admitted.

We choose this over mapping every final-answer validation failure directly to `REFUSED_NO_EVIDENCE` inside the answer adapter because operators need to distinguish no evidence, schema failure, citation-binding failure, adequacy failure, policy denial, and model-call denial without reading raw provider output. We choose attempt-owned diagnostics over Delivery-layer trace reconstruction because the Control Plane produced the failure facts and Observability should consume those facts rather than infer workflow semantics from logs.

Final-answer repair retry remains inside this attempt boundary. It is a bounded phase after normalization failure and before admission, not a new ReAct action, not an Orchestrator loop iteration, and not a Delivery retry. This preserves Orchestrator ownership of terminal workflow semantics while allowing the final-answer module to repair model-output contract drift using the same evidence, context, policy, trace, and validation boundaries as the original attempt.

Repair eligibility is deterministic and conservative. Schema failures, citation binding failures with available allowed citation refs, and final-answer adequacy failures without safety failure may receive one repair retry. Safety failure, policy denial, no accepted evidence, citation binding without allowed refs, and exhausted retry budget are not repairable. The repair request may carry stable validation codes, bounded field paths, the original question, allowed citation refs, and the previous response parsed as JSON when available, but must not turn untrusted prior conversation into evidence or ask the model to invent missing citations.

When the initial candidate fails validation but the repair succeeds, the failed candidate remains visible as ordinary trace history and a validation-only LLM interaction capture, but it is not returned as a `WorkflowStageFailureDiagnostic` because the `model_answer` stage did not finally stop. If the repair also fails or repair is not eligible, the final unrepaired validation failure is returned as the stage failure diagnostic.

Repair request payloads are intentionally small and stable. They may include the original question, the same accepted evidence text and allowed citation refs, admitted follow-up context summary marked as non-evidence, stable validation error codes, bounded violation codes and field paths, violation count, previous response JSON when it parses, previous parse error code when it does not parse, and the required output contract/function schema. They must not include free-form validator reasons, raw stack traces, provider envelopes, unadmitted conversation turns, extra retrieval results, invented citation lists, or previous content from a safety-failed candidate.

Each repair retry is a new governed model call. It must pass the same `BEFORE_MODEL_CALL` policy evaluation as the initial generation and emit its own policy decision plus model request and response trace projections with repair-attempt metadata. If policy denies the repair call, the attempt maps to `POLICY_DENIED`; the denial is not reported as a final-answer validation failure.

The module should expose an internal typed attempt status before outcome mapping. The status set starts with `admitted`, `policy_denied`, `schema_failed`, `safety_failed`, `citation_binding_failed`, `final_answer_adequacy_failed`, `validation_failed`, and `model_error`. Repairability is derived from status, accepted-evidence facts, allowed citation refs, and retry budget rather than modeled as a separate status.

Final Answer Attempt consumes admitted follow-up context from Controlled ReAct run state as non-evidence context. The same typed `conversation_context` should also be available to intent resolution and planning so references such as "these", "that product", or "the previous answer" can influence retrieval and action selection before the final-answer model sees evidence. This context must not be mixed into Answer Evidence Context or treated as citation support.

The first repair-retry implementation is default-on for eligible validation failures, capped at one retry, and does not add a new Agent Contract or manifest option. Existing model-call policy remains the control surface for cost and role governance. If operators later need direct tuning, repair retry limits should be exposed through Workflow Stage Configuration rather than widening the first implementation's YAML surface.
