# Blue/Green Releases Drain Before Atomic Switch

Accepted.

[FRAME | HIGH] Initial production keeps a stable Gateway outside the Blue and Green application slots and deploys each candidate as an immutable image and configuration set into the inactive slot. A locked, expand-only database migration runs first; the candidate API and application surfaces then start while its Run Executor and Knowledge Worker remain `STANDBY`. The candidate must pass production-dependency readiness and isolated smoke checks before any production ownership changes. Dashboard, Operator Chat, API, OIDC callback, and SSE always route to one slot together and cannot be switched independently.

[FRAME | HIGH] The active Run Executor enters `DRAINING`, stops new claims, and receives at most 150 seconds to finish its claimed Attempts while the active API continues admitting bounded requests into the PostgreSQL queue. Failure to reach zero claims aborts the release and restores the old Executor without changing Gateway routing. The versioned queue and execution contract must be compatible in both directions across the rollback pair: the candidate Executor accepts requests admitted by the old API, and the old Executor accepts requests admitted by the candidate API. After a successful drain, Gateway routing changes atomically and the candidate obtains the sole PostgreSQL role-activation lease with a higher fencing epoch; stale slots cannot claim work or commit terminal state. External checks through the stable origin must then prove OIDC, submission, progress, terminal state, and S3-First Artifact Finalization.

[FRAME | HIGH] The candidate soaks for 30 minutes before old compute is stopped. The old image, configuration, migration compatibility, and rollback procedure remain available until the later of 24 hours after switching and the end of one complete 09:00-18:00 `Asia/Shanghai` support window. Rollback routes Gateway back, drains the candidate, and grants the old slot a new activation epoch; it never runs a database down migration. Contract migrations and destructive cleanup occur only after the rollback window closes in a separate maintenance release.
