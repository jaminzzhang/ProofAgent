# MCP Tool Source Adapters Behind Tool Gateway

Accepted.

Proof Agent will support real MCP tool calls by implementing MCP stdio and HTTP clients as **MCP Tool Source Adapters** behind Tool Gateway. The first supported **MCP Transport Adapter** set includes both stdio and HTTP rather than treating HTTP as a later contract placeholder. An LLM may propose a tool call through a Harness-normalized action proposal, but it must not execute provider-native tool calls or connect directly to MCP servers. This keeps function/tool calling as a structured intent, while Tool Gateway remains responsible for allowlisting, parameter validation, authorization, approval pause/resume, execution, trace, receipt, and structured tool-result normalization.

We chose this over model-native MCP execution because Proof Agent's product boundary is governed enterprise execution, not raw Agent autonomy. MCP supplies a standard connection protocol for external tools; it does not replace Tool Contract, Agent Tool Binding, PolicyEngine, approval state, or observability contracts.

The runtime adapter follows the **MCP SDK Protocol Boundary**. V1 uses the official MCP Python SDK for protocol operations such as initialize, `tools/list`, and `tools/call`. It does not execute governed tools through `langchain-mcp-adapters` or LangChain Tool objects, because schema mapping, proposal scope, authorization, approval, result validation, retry policy, trace, and receipt semantics must remain owned by Proof Agent contracts and Tool Gateway.

V1 also follows an **MCP Tools-Only V1 Boundary**. It supports MCP initialize, `tools/list`, and `tools/call` only. MCP resources, prompts, sampling, elicitation, and other protocol capabilities are excluded from V1 because Proof Agent already has separate governed Knowledge Source, Evidence Admission, Harness Control Prompt, and Workflow Stage Prompt Configuration boundaries.

MCP runtime connections use **Run-Scoped MCP Session** lifecycle. Discovery and validation create short-lived sessions and close them when the operation completes. Runtime may reuse one session for one Tool Source within one active governed run, but must close it at run completion, approval pause, or failure. Sessions are not pooled across runs, tenants, caller contexts, or Published Agent Versions. Approval pause must not hold an MCP session open; approval resume re-initializes the MCP session and executes only from the frozen approved parameters.

MCP Tool Discovery may inspect a server's available schemas, but discovery does not grant Agent authority. Operators must use **Curated MCP Tool Import** to convert selected discovered schemas into governed Tool Contracts, then bind those contracts to Agents through Agent Tool Bindings. Proof Agent excludes automatic exposure of an MCP server's full tool catalog because broad runtime catalogs weaken risk classification, parameter bounds, approval policy, and evaluation coverage.

Curated MCP Tool Import freezes the selected discovered schema as an **MCP Tool Contract Snapshot**. Runtime validation uses the frozen snapshot captured by the Tool Contract and any Published Agent Version binding, not the latest live MCP server schema. If a later MCP server schema changes, operators may refresh and import a new Tool Contract revision; existing Published Agent Versions continue using their captured contract and fail closed if the live server no longer satisfies it.

MCP binding requires validation before publication. **MCP Tool Source Validation** must initialize the configured stdio or HTTP transport, authenticate only through allowed credential references, list tools, confirm the imported tool still exists, and check live-schema compatibility with the frozen MCP Tool Contract Snapshot. **MCP Agent Tool Publication Validation** must then verify that every bound MCP Tool Contract references an active Tool Source, has complete parameter, result, summary, risk, approval, and redaction rules, and satisfies MCP Action Tool Governance for state-changing tools. Failed validation blocks publication.

MCP Tool Sources have two configuration entry points with different governance. **Dashboard-Managed MCP Tool Sources** are the production reusable path in the Shared Asset Library, with lifecycle, validation, operator permissions, and configuration audit. **Package-Local MCP Tool Sources** are allowed for public examples, tests, local development, and reviewable package-scoped execution; they may declare stdio commands or local HTTP endpoints, but must not contain inline secrets and must still pass MCP Tool Source Validation before execution or publication. Package-local MCP definitions do not automatically become shared assets.

HTTP MCP V1 follows an **MCP HTTP Credential Boundary**. It supports unauthenticated endpoints and static environment-variable credential references such as bearer tokens or API-key headers, with redaction in configuration, trace, receipt, and error paths. It excludes OAuth browser or device flows, refresh-token management, per-user delegated authorization, and runtime user authorization prompts because those require a separate credential lifecycle and identity design.

MCP tool results follow **MCP Tool Result Classification**. The default result path is an Observation Record in the Controlled ReAct Loop. A result may support a customer-facing business claim only when the Tool Contract marks it as an authoritative read result and Tool Gateway has produced an Authorized Tool Result through authorization, schema validation, source labeling, and redaction. MCP results do not become Accepted Evidence and must not bypass Evidence Admission.

Planner-visible MCP result summaries follow **MCP Tool Summary Projection**. The MCP server and MCP adapter may return and normalize full payloads for the observation truth layer and audit replay, but they do not decide what the planner sees. The Control Plane extracts the planner-visible summary only from Tool Contract `summary_fields` after result-schema validation and redaction.

MCP runtime audit uses **MCP Tool Audit Projection**. Trace may record tool source id, provider, transport, Tool Contract id, contract snapshot digest, server label, endpoint host or stdio command digest, credential environment-variable reference name, approval id, policy decision, parameter keys, redacted parameter digest, result-schema validation status, summary fields used, latency, retry count, failure class, and action-tool side-effect or idempotency digests. Governance Receipt renders only a compressed human-readable summary of governed MCP tool use, approval, execution, result classification, and failure status. Neither trace nor receipt may include secret values, raw credential-bearing command or endpoint values, complete raw MCP payloads, or unbounded model-visible content.

MCP execution failures fail closed. Transport timeouts, HTTP 401/403, transient server failure after bounded retry, stdio process exit, schema mismatch, missing credentials, or unsafe responses produce an **MCP Tool Execution Failure** fact in trace and, when the Controlled ReAct Loop can continue, an Observation Record that returns to plan. Read-only MCP tools may use a bounded single retry for transport timeout or transient 5xx responses, and that retry consumes tool budget. Write or action MCP tools do not auto-retry in V1; a later retry policy must require idempotency and explicit governance.

V1 may support state-changing MCP tools only under **MCP Action Tool Governance**. These tools are not enabled by default, must be imported and bound explicitly, must declare `read_only: false`, `requires_approval: true`, an `idempotency_key` parameter, and a side-effect classification, and must not auto-retry. Their results are action confirmations or Observation Records, not Accepted Evidence or ordinary Authorized Tool Results for factual claim support. Customer-facing or domain-specific product surfaces may still prohibit transactional execution even when the underlying MCP capability exists.

Runtime model context uses **Effective Tool Proposal Scope**, not the full Agent tool catalog or live MCP server catalog. Even after MCP tools are imported and bound, the planner sees only a bounded tool summary admitted for the current run, stage, caller context, policy precheck, and budget. Full frozen parameter and result schemas are used for proposal validation and Tool Gateway execution rather than being blindly dumped into every planning prompt.

Implementation should follow governance dependencies instead of starting with an unconstrained runtime demo. The slices are: first contracts and store records for MCP Tool Sources, credential references, snapshots, and tool contract metadata; second stdio/HTTP discovery and Curated MCP Tool Import; third Tool Source and Agent publication validation; fourth Tool Gateway runtime execution, result validation, summary projection, audit projection, and failure semantics; fifth deterministic stdio/HTTP fake MCP servers and tests for approval, denial, failure, schema drift, action idempotency, and audit output.

Verification must cover the governance boundary, not only a successful MCP call. Required tests include secret rejection and env-ref redaction; package-local and Dashboard-managed Tool Source validation; discovered-but-not-imported tools being unavailable for Agent binding; MCP Tool Contract Snapshot digest capture; schema drift blocking publication or failing closed at runtime; stdio and HTTP read tools executing only through Tool Gateway; denied parameters and unregistered tools being rejected before MCP calls; approval pause/resume and denial behavior; result-schema mismatch; Tool Contract `summary_fields` as the only planner-visible summary source; raw MCP payload exclusion from planner summaries; action tool publication failure without `idempotency_key`; no automatic action-tool retry; trace MCP Tool Audit Projection fields; Governance Receipt compressed summaries without raw args, raw results, or secrets; and evaluation gates for expected tool calls, approval, failure, and raw-payload leakage prevention.
