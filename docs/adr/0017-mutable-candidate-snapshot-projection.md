# Mutable Candidate Knowledge Source Snapshot Projection

A Candidate Knowledge Source Snapshot is a mutable Source Draft projection rather than a new immutable version after every document or worker state transition. A lightweight Knowledge Source Draft Version change token advances when source configuration or candidate-snapshot membership changes, while worker leases, retry counters, and rejected uploads do not create draft churn. Foundation Knowledge Source Publication Validation captures that token, and snapshot freeze rejects stale validation before freezing the current eligible READY revision set into a new Frozen Knowledge Source Snapshot. This keeps ingestion and batch processing from generating noisy intermediate versions while preserving explicit publication, auditability, and rollback at the published-snapshot boundary.

The first publication slice may persist an explicitly labeled `foundation` validation level that checks READY revision membership, compatible artifact references, pending required reingestion, and Knowledge Source Draft Version freshness. It may freeze development-stage READY snapshots for subsequent document-routing work, but it must not imply production-ready local-index validation until routing-model configuration tests, smoke retrieval, and citation resolution validation exist.

A frozen or published local-index snapshot is an immutable manifest of document-revision identities, trace-safe routing metadata, and reusable artifact references. Snapshot freeze and publication do not copy artifact directories and do not build a merged snapshot index. Runtime document routing selects bounded manifest entries and loads their referenced revision artifacts, while cleanup preserves any artifact still referenced by a retained snapshot.

The mutable candidate projection is derived from managed document state rather than persisted as a second `candidate.json` record. Store operations advance the Knowledge Source Draft Version when candidate membership changes. Snapshot freeze derives the candidate again under the store lock, rejects stale validation, atomically writes the immutable snapshot manifest, and then atomically advances the latest snapshot pointer.

The publication-foundation slice keeps the current managed `KnowledgeDocument` active-revision projection rather than introducing replacement uploads, archive, cleanup reference checks, and full revision-history storage at the same time. Snapshot manifests still freeze `document_id`, `revision_id`, and artifact reference from the first version, so published snapshots remain immutable and the later revision-history slice does not need to migrate retained snapshot manifests.

Publication foundation and multi-document runtime routing ship as two consecutive slices joined by the `local_index.snapshot.v2` manifest contract. The first slice owns the Source Draft token, derived candidate summary, foundation validation record, immutable manifest freeze, latest snapshot pointer, and management APIs. The second slice consumes that manifest for bounded document routing, per-revision artifact load, fail-closed selected-document search, and trace-safe routing summaries. Keeping the state-machine and query-time changes separate preserves focused verification without weakening the shared contract.

A foundation-validated snapshot is a Frozen Knowledge Source Snapshot, not a production publication. The Source records it through `latest_snapshot_id` for preview and routing-smoke development only. A separate `published_snapshot_id` remains unchanged until full routing-model, smoke-retrieval, and citation-resolution validation exists and the explicit Knowledge Source Publication action promotes a validated immutable snapshot for Agent binding. This avoids relying on every binding path to remember an extra `validation_level` check.

Snapshot freeze is idempotent for one unchanged Knowledge Source Draft Version and candidate digest after successful foundation validation. The deterministic snapshot identity intentionally excludes the validation-record identity, so repeated validation of an unchanged candidate still reuses the existing matching Frozen Knowledge Source Snapshot rather than creating noisy duplicate versions after client retry or replay.
