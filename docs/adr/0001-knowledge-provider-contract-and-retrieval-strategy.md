# Knowledge Provider Contract And Retrieval Strategy

The first-stage Agent Contract introduced `knowledge.provider`, provider-specific `knowledge.params`, and a required top-level `retrieval` section instead of the earlier `knowledge.path` shape. The 2026-05-30 Shared Knowledge Source Binding amendment below supersedes inline Agent provider configuration: Knowledge Sources now own providers, while Agents bind shared Sources.

First-stage provider names were `local_markdown`, `local_vector`, `remote_search`, and `pageindex`. Agentic RAG is a governed retrieval strategy selected through the top-level `retrieval` section, not a Knowledge Provider and not a separate business workflow template.

## 2026-06-01 Amendment: Knowledge Hub V1 Provider Set

ADR-0016 supersedes the first-stage provider set. Knowledge Hub V1 keeps `local_markdown` for deterministic development fixtures, uses `local_index` for production local indexed knowledge, and uses trusted remote adapters such as `http_json` for remote retrieval. `pageindex` and `local_vector` are removed from the target provider set rather than retained as hidden compatibility entries; remaining references to PageIndex or local vector behavior in this ADR are historical context.

## 2026-05-12 Amendment: Superseded PageIndex Agentic Retrieval

This amendment is superseded by ADR-0015 and ADR-0016. It remains here only as historical context for why Agentic RAG is modeled as a governed retrieval strategy rather than as an answer-generating provider.

The `pageindex` provider integrates a self-hosted PageIndex deployment through the retrieval API. Proof Agent uses PageIndex as a remote evidence source, not as the final answer generator. In `retrieval.strategy: agentic`, the Harness emits policy-gated `retrieval_plan` and `retrieval_step` events, delegates the remote reasoning-based retrieval step to PageIndex, then evaluates the returned candidate evidence locally before any model call.

## 2026-05-30 Amendment: Shared Knowledge Source Binding

Dashboard-managed reusable Knowledge Sources supersede direct Agent-to-provider binding as the target configuration model. A Knowledge Source owns its provider and provider-specific parameters. An Agent Contract binds one or more Knowledge Sources through an Agent Knowledge Binding Set and configures governed retrieval behavior without selecting providers. At Agent publication, the Published Agent Version captures a resolved Knowledge Source snapshot identifier for every binding so source publication cannot silently change an existing Agent version.

Shared Knowledge Source resources live under `/api/config/knowledge-sources`, with collection operations for list, create, and filter; Source detail and Draft update; archive and restore; nested documents and versions; validation and publication; remote connection testing and retrieval preview; and import-export actions. Agent Draft configuration stores `knowledge_bindings[]` and Agent-level blended-retrieval settings only. Source-owned provider parameters are resolved through shared assets, and Published Agent Versions record the resolved Source snapshot or configuration version plus resolved binding settings.

Because Agents are not yet deployed, this is a direct breaking migration rather than a compatibility layer. Loader, Dashboard, examples, fixtures, and tests move together to Source-owned provider configuration plus Agent `knowledge_bindings[]`. The new loader rejects inline Agent `knowledge.provider + params`; it does not dual-read, auto-wrap, or preserve a legacy runtime provider path.

Knowledge Source lifecycle is separate from ingestion readiness and pinned Published Agent execution. An active source permits editing, publication, and new Agent binding. Archiving is reversible: it blocks new binding and new Agent publication, preserves retained snapshots and configuration versions, and leaves existing Published Agent Versions able to execute against their pinned references. Dashboard shows affected Published Agents before and after archive. Restore returns the source to active state without updating any Agent automatically. Physical deletion is allowed only after retained snapshots, Published Agent references, and audit-retention requirements no longer require the source.

Knowledge administration audit and retrieval runtime facts remain separate trace-safe records. Configuration audit records actor, timestamp, affected Source or Agent, prior and resulting version identifiers, document intake and replacement, retry, document archive, Source publication, Source archive and restore, remote verification, Agent binding changes, retrieval override changes, and explicit Source upgrades. Trace, Governance Receipt, and RunStore record runtime-resolved source versions, routed sources and local document revisions, provider call status, degraded retrieval, upstream revision observations, WRRF ordering, exact-dedup provenance, admission scores, citations, and context-budget truncation. Neither record stores raw document content, credential values, or complete remote responses.

Runtime retrieval planning records a bounded two-stage summary. `binding_candidates[]` captures each Agent-bound Source with source id, alias, tags, lifecycle, resolved published version, failure mode, and fusion weight. `selected_bindings[]` captures the selected bindings with source id, selection reason, routing score or order when available, and required or advisory mode. Local indexed retrieval additionally records `document_candidates[]` and `selected_documents[]` without raw content. Provider calls record success or failure, latency, candidate count, stable error code, and upstream revision observation when available. RunStore and Dashboard Run Detail preserve the full summary; Governance Receipt keeps a compressed summary.

An unpinned Draft Agent binding resolves to the latest published Knowledge Source snapshot or configuration version and Dashboard shows that currently resolved version. A Published Agent Version remains pinned to the source snapshot or configuration version and resolved binding settings captured during publication. When a source publishes a newer version, Dashboard marks older bindings as upgradeable; an administrator applies the update to a Draft Agent, reruns validation, and publishes a new Agent version. A `mutable_external` remote source pins only its immutable Proof Agent-managed connection configuration version and records observed upstream revision data when available, so Dashboard must continue to warn that exact corpus replay is unavailable.

An Agent Knowledge Binding may override only bounded retrieval behavior: provider retrieval `top_k`, binding fusion weight, binding failure mode, and source-routing metadata hints. Missing values inherit the Knowledge Source defaults, and publication captures the resolved binding configuration. The binding cannot override provider endpoint, credentials, index or namespace, local index ingestion parameters, or admission scorer; those remain Knowledge Source-owned.

Multi-source retrieval is governed blending, not an implicit provider fan-out. The Harness routes a query to a bounded subset of eligible Agent Knowledge Bindings, executes provider-specific retrieval for each selected source, normalizes and merges Candidate Evidence, then applies the existing Control Envelope evidence admission and citation rules. This keeps provider lifecycle, ingestion, credentials, and reusable asset management at the Knowledge Source boundary while preserving Retrieval Strategy and evidence admission as Control Envelope concerns. The previous inline `knowledge.provider` manifest shape is removed by direct migration rather than retained as a compatibility path.

Knowledge Source Routing first filters binding and source metadata, then uses an Agent-specific routing model configuration to select a bounded subset. Knowledge Source Selection Budget defaults to 3 and is configurable from 1 through 8. Empty routing returns no evidence rather than querying every binding, and uncertain routing may widen only within the configured budget. This Agent-level source fan-out limit is distinct from one local index source's Knowledge Document Selection Budget and from final evidence `top_k`.

Selected Knowledge Sources may use heterogeneous Knowledge Provider Adapters, including local indexes, trusted remote adapters, and future technology-specific integrations. Each adapter translates its backend-specific response into the provider-neutral Candidate Evidence contract. Cross-source fusion operates on normalized candidates, the Control Envelope admits or rejects evidence, and only Accepted Evidence is assembled into final-answer LLM context. Raw provider responses and rejected candidates must not bypass evidence admission.

Candidate Evidence carries trace-safe identity fields: evidence id, source id, source version id, binding id, provider name, optional document id, optional revision id, optional chunk id, citation, provider-native score, fusion rank, admission score, and allowlisted metadata. Citation is mandatory before final-answer LLM context assembly. When exact deduplication merges candidates, the merged candidate preserves one contribution per source with source, source version, binding, provider, available document or chunk identifiers, provider-local rank, provider-native score, binding fusion weight, and citation.

V1 keeps Knowledge Provider adapters inside a trusted code boundary. Registry entries become adapter descriptors that declare a configuration schema, Dashboard form metadata, and capabilities such as `retrieve`, `health_check`, `snapshot_pin`, and `admission_score`. A generic trusted `http_json` adapter covers validated remote retrieval APIs, while specialized stacks may add typed adapters through code installation. Dashboard configuration stores provider parameters and environment-variable credential references only; it does not accept executable adapter scripts.

The `http_json` adapter provides a versioned default Remote Retrieval Protocol for the zero-mapping path and also supports versioned declarative request and response mappings for non-standard remote APIs. Request mappings may project only whole-value placeholders `${query}`, `${top_k}`, and `${upstream_revision}` into query parameters and JSON body fields while preserving source types. Endpoint and URL path remain static published configuration; headers may be static non-sensitive values or environment-variable references with an optional static prefix. Response mappings project remote JSON into normalized Candidate Evidence and optional revision-observation fields. These mappings are configuration, not executable code: they cannot use string interpolation, dynamic URL paths, loops, conditions, functions, network callbacks, scripts, raw secret values, raw LLM-context injection, or evidence-admission bypass. Published remote source configuration versions capture the selected protocol version and mappings for validation, audit, and rollback.

V1 response mapping uses JSON Pointer only: one pointer selects the result array, relative item pointers select id, content, citation, structured `source_ref` fields such as document id, page, and chunk id, provider-native score, and metadata, and an optional top-level pointer selects upstream revision observation. JSONPath wildcards, filters, recursive queries, functions, concatenation, calculations, and direct mapping into evidence admission score are excluded. Remote Knowledge Source Verification resolves the mapping against a health-check sample response and fails closed unless the result pointer is an array and normalized candidates contain content plus either citation or an adequate structured `source_ref`. Trusted adapter code formats an adequate `source_ref` into citation using deterministic internal rules; Dashboard administrators cannot author arbitrary citation templates. Candidates lacking both usable citation and adequate `source_ref` remain traceable but are inadmissible and excluded from LLM context. Evidence admission score may come only from an approved calibrated adapter descriptor or approved admission scorer.

Local indexed evidence uses stable internal citations such as `knowledge://source/{source_id}/document/{document_id}/revision/{revision_id}#page=12`, or a Markdown section anchor, without exposing storage paths. Dashboard Run Detail and retrieval preview resolve these citations through permission-protected audited previews. Customer-facing output projects safe source name plus page or section and omits internal revision identifiers and storage paths. Remote citation URLs become clickable only after protocol and domain allowlist validation; otherwise the citation remains visible as non-clickable source text. Citation preview access and original-document download are separate audited actions.

Customer-facing answers use short inline citation markers such as `[1]` and a Sources list at the end of the answer. The Sources list maps each marker to safe source name, page or section, and safe document title, merging repeated references to the same safe source location. It omits confidence, provider details, internal source or revision identifiers, and mutable-external replay warnings. If no Accepted Evidence exists, the model must not invent citations and must follow insufficient-evidence refusal behavior.

When no Candidate Evidence becomes Accepted Evidence, the runtime follows the governed insufficient-evidence or refusal path. It skips free-form final-answer generation or uses only a constrained refusal template without evidence context, and the customer response does not include a Sources list. Run Detail shows whether routing, provider retrieval, fusion, citation enforcement, admission, or provider failure caused the absence of evidence. Receipt records zero Accepted Evidence, candidate count, refusal reason code, and whether provider failure contributed. Advisory provider failures may still answer when remaining selected bindings provide Accepted Evidence; selected required provider failures prevent answering.

Every published remote source captures an immutable Proof Agent-managed connection configuration version. If its adapter supports `snapshot_pin`, publication also records an immutable upstream corpus revision and produces a pinned remote source. If the upstream stack cannot pin a revision, the source remains explicitly `mutable_external`: it may be bound and queried, but Dashboard, validation artifacts, Trace, Governance Receipt, and RunStore must not imply exact historical replay. Each remote retrieval records an upstream revision, etag, or observation timestamp when available.

A remote adapter must declare `health_check` and pass Remote Knowledge Source Verification before its source can be published for production bindings. Verification covers connectivity, authentication, target index or namespace existence, and response normalization, then records verification time, adapter name, immutable connection configuration version, and available upstream revision observation. Adapters without `health_check` remain preview-only. Mutable external verification is valid for 24 hours by default; stale verification warns operators and blocks new Agent publication or rebinding until refreshed without immediately interrupting already-published Agent execution.

Source publication validation is bound to one Source Draft version and becomes stale after relevant configuration changes. Remote Source validation requires current health-check verification, authentication, target index or namespace validation, response normalization, and an editable smoke query proving normalized candidate evidence, citation or adequate structured `source_ref`, and available upstream revision observation. Adapters without `health_check` remain preview-only and cannot publish for production bindings.

V1 cross-source fusion uses Weighted Reciprocal Rank Fusion over each selected source's provider-local result ranks and resolved source weights. Provider-native relevance scores remain available for source-local ordering and audit trace, but the Harness does not sort heterogeneous providers by raw score. The resulting cross-source fusion rank controls bounded candidate ordering before evidence admission; it does not itself create Accepted Evidence or bypass Evidence Threshold.

Before final fusion ordering, V1 performs exact cross-source evidence deduplication only when canonical citation or trusted-formatted structured source reference plus normalized content hash match exactly. A merged candidate preserves every contributing Knowledge Source, Agent Knowledge Binding, and citation, and its WRRF contributions combine. V1 does not collapse candidates on content hash alone and does not perform semantic-similarity deduplication, because those approaches may erase independently attributable evidence.

Duplicate retrieval hits affect WRRF ordering but do not boost evidence admission score. When an approved admission scorer is configured, it evaluates the merged normalized chunk once. Otherwise, the merged candidate uses the minimum available calibrated evidence admission score from its contributing sources. Contributors without calibrated admission scores remain traceable but do not participate in score aggregation; if no valid admission score remains, the merged candidate is inadmissible.

Each Agent Knowledge Binding owns its positive fusion weight, defaulting to 1.0. The shared Knowledge Source does not own a global fusion weight because the same source may have different importance for different Agents. Published Agent Version snapshots capture the resolved binding weight used for validation and execution.

Each Agent Knowledge Binding also owns a retrieval failure mode, defaulting to `required`. A selected required binding failure fails the whole retrieval. An explicit `advisory` binding failure permits degraded retrieval from remaining selected bindings, but it is never silent: Trace, Governance Receipt, and RunStore record the failed advisory binding summary, and remaining evidence still passes through ordinary fusion, admission, and citation enforcement.

The target evidence contract separates three values: provider-native relevance score for source-local ordering and trace, cross-source fusion rank for bounded candidate ordering, and evidence admission score for Control Envelope threshold evaluation. Evidence admission score is a conservative normalized value from 0 through 1 supplied only by an adapter with reliable backend-specific mapping or by an approved admission scorer. A candidate without an evidence admission score remains traceable but cannot enter final-answer LLM context.

Final-answer LLM context assembly iterates admitted, cited candidates in cross-source fusion rank order and stops when either Agent-level budget is exhausted. The chunk budget defaults to 12 and is configurable from 1 through 40. The approximate evidence-context token budget defaults to 6000 and is configurable from 500 through 20000. V1 does not reserve fixed per-source context quotas. Trace records how many already-admitted candidates remained outside LLM context because of budget truncation.

Each included chunk is projected into a fixed prompt-safe context item containing evidence id, source label, citation label, content, a low/medium/high confidence band derived from admission score, source type, and context rank. The final-answer LLM does not receive provider-native score, numeric fusion rank, internal source or version identifiers, revision ids, raw provider payload, or original file paths.

The current `EvidenceChunk.score` field, validator, graph state, RunStore projection, Governance Receipt projection, providers, fixtures, and tests use one overloaded score. This changes directly in the same breaking cutover as Agent knowledge configuration: Candidate Evidence uses optional `provider_native_score`, WRRF produces `fusion_rank`, and the Control Envelope evaluates optional `admission_score`. The old `score` field is removed rather than preserved as a single-provider alias, and Trace, RunStore, and Governance Receipt record the three meanings separately.

Implementation proceeds in this order: first migrate contracts and loader to `knowledge_bindings[]`, Source-owned provider configuration, and split evidence score fields; then add the Knowledge Source store and configuration API; then implement `local_index` upload, worker, artifact cache, runtime load, and snapshots; then implement multi-source runtime routing, provider calls, WRRF, deduplication, admission, and context assembly; then build the Dashboard `/knowledge` workspace, Source detail tabs, Agent binding editor, and Run Detail projections; finally migrate fixtures and tests and remove the legacy inline `knowledge.provider + params` path.
