# Agent Contract Stage, Capability, and Trace Capture Contract

Accepted.

Proof Agent will make a direct breaking Agent Contract migration for React Enterprise QA configuration rather than carrying dual-read compatibility. Public Workflow Template configuration uses **Workflow Template Stage** terminology through `workflow.stages[]`; each stage override uses `id` to reference a registered stage. Loaders reject legacy `workflow.nodes[]`, `node_id`, and `stage_id` fields. `workflow.stages[]` stores only explicit **Workflow Stage Configuration Override** entries; omitted available stages use descriptor defaults when building the **Effective Workflow Stage Configuration**. Implementation code that models Agent Contract, manifest loading, validation, Workflow Template descriptors, configuration APIs, Dashboard helpers, examples, fixtures, and tests should use stage terminology. Agent Configuration API request and response fields move from `nodes` to `stages`, Dashboard helpers and UI labels use stage terminology, and Dashboard fixtures and tests migrate in the same cutover. Slice 1 starts from tests that lock accepted and rejected schema shapes before loader/API/Dashboard implementation, and every slice follows small-step verification: targeted test, minimal implementation, targeted verification, then broader regression. Proof Agent will not ship a legacy Agent Contract migration script or codemod; source-controlled examples and fixtures are edited directly, while generated local Agent Configuration Store data is cleared and recreated. Runtime Plane graph node terminology may remain where it describes actual LangGraph or runtime graph mechanics. `ReActWorkflowNodes` renaming or splitting is deferred to the later Workflow Template Execution consumption slice.

Stage prompt configuration remains bounded business context, not control prompt authority. The first allowed `workflow.stages[].prompt` fields are `business_context`, `task_instructions[]`, and `output_preferences[]`. Proof Agent rejects `system_prompt`, `developer_prompt`, `raw_prompt`, and `role_guidance` because governed system and developer prompt authority belongs to the Harness. Stage context configuration uses descriptor-allowed boolean keys under `workflow.stages[].context`: missing keys inherit descriptor defaults, explicit `true` includes the option, and explicit `false` disables it only when the option remains inside the effective allowlist.

Agent Contract YAML will use top-level `capabilities:` for optional governed capability domains that affect stage availability. `capabilities.tools.enabled` controls the paired `tool_review` and `tool` stages, and `capabilities.memory.enabled` controls the memory stage for templates that include one. Legacy top-level `tools` and `memory` fields are rejected. Every template-relevant capability domain must explicitly declare `enabled`; missing domains or missing `enabled` values are validation errors, not implicit `false`. Disabled capability domains must be clean in published YAML and may not carry active config such as `tools.file`, memory provider, or memory scopes. When enabled, tools must resolve at least one valid Tool Contract, and memory must satisfy provider-specific readiness rules. Knowledge bindings remain a separate `knowledge_bindings[]` Agent Knowledge Binding domain because React Enterprise QA retrieval stages are required evidence behavior rather than optional stage availability.

Agent Publication resolves and freezes the **Workflow Stage Availability Set**, the **Effective Workflow Stage Context Option Allowlist**, and the full **Published Effective Workflow Stage Configuration Snapshot** into the Published Agent Version alongside the original overrides and Workflow Template Descriptor Version. Published Agent Versions must not dynamically reinterpret stage availability or descriptor defaults from mutable template code. This intentional redundancy keeps historical run explanation stable even if descriptors evolve later.

Run trace records **Workflow Stage Configuration Trace Summary** by default: stage id, configured prompt field names, bounded counts and lengths, context option names, redaction status, and Published Agent Version or effective snapshot reference. Full prompt text, selected context payloads, and intermediate Workflow Stage Result details are not ordinary production trace content. A validation/test-run request may enable **Workflow Stage Trace Capture Mode** full capture for verification, but production and customer-facing runs are forced to summary-only trace behavior. Full capture must still exclude raw chain-of-thought, secrets, raw evidence content, raw tool payloads, complete provider responses, and Runtime Plane state dictionaries.

Full capture output is a **Sensitive Validation Capture Artifact** rather than ordinary trace content. Access requires `agent.validate` or stronger configuration permission, ordinary run viewers and customer support viewers cannot see it, ordinary receipts and exports exclude it, and Dashboard reveals it only through explicit validation detail UI. Sensitive validation capture uses a default 7-day TTL; `retain_for_audit` may extend retention only through an authorized validation/test-run request. Expiring full capture artifacts must not change the ordinary summary trace, receipt facts, validation outcome, or governed run result.

This ADR extends ADR-0027's stage terminology cutover and supersedes ADR-0022's public `workflow.nodes[]` shape while preserving ADR-0022's Harness prompt authority boundary.
