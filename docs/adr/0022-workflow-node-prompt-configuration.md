# Workflow Stage Prompt Configuration

> Superseded in part by [ADR-0028](0028-agent-contract-stage-capability-and-trace-capture-contract.md): public `workflow.nodes[]` configuration is replaced by `workflow.stages[]`; the Harness prompt authority boundary remains valid.
>
> Superseded in part by [ADR-0029](0029-deterministic-react-baseline.md): the long-term deterministic regression baseline is now the deterministic path through `react_enterprise_qa`, while `enterprise_qa` may remain only as a compatibility path.

Proof Agent adds Workflow Stage Prompt Configuration for the `react_enterprise_qa` template so Agent owners can provide richer business context for governed ReAct execution without editing workflow topology or replacing Harness-owned control prompts. The active public contract uses structured per-stage Prompt and context settings inside `workflow.stages[]` because Agent owners need to understand and tune the task context around `plan`, `clarification`, `retrieval_review`, `retrieval`, `model_answer`, `tool_review`, `tool`, `memory`, and `response` stages, while the Control Envelope must remain the source of workflow order, policy gates, validators, Tool Gateway behavior, trace semantics, and model output contracts.

V1 will not support free-form runtime graph editing, arbitrary stage creation, edge editing, stage disabling, shared Prompt templates, or prompt-defined workflow behavior. Full Workflow Stage Prompt Configuration is scoped to `react_enterprise_qa`; `enterprise_qa` may expose a read-only workflow summary but is only a compatibility path after ADR-0029. The Agent Contract stores only stable public stage ids, not LangGraph internal node names.

The backend Workflow Template Registry owns a Workflow Template Descriptor and Workflow Template Descriptor Version for each registered template. The descriptor is the source of truth for the Dashboard Workflow Relationship Map, stage order, branch relationships, governed handoff points, editable Prompt fields, allowed context options, stage input and output summaries, and model-bearing versus non-model governed stage classification. The Dashboard renders this descriptor instead of hard-coding workflow semantics.

`workflow.stages[]` stores structured Agent-owner-editable business Prompt settings and selected context options. Supported Prompt fields are intentionally narrow: `business_context`, `task_instructions[]`, and `output_preferences[]`. Context options are descriptor allowlists per stage, such as agent purpose, recent conversation summary, bound Knowledge Sources, bound Tools, policy outline, retrieval intent, evidence summary, citation requirements, approval requirements, memory scope, or response disclosure policy. `system_prompt`, `developer_prompt`, raw chain-of-thought requests, policy overrides, direct tool execution instructions, validator bypasses, and raw document or raw trace injection are not supported.

At runtime, stage Prompt configuration becomes a Business Context Addendum appended to the governed stage's structured model context after Proof Agent's Harness-owned control prompt and Structured Control Context. It never replaces the Harness-owned `ModelRole.SYSTEM` prompt, JSON contract instructions, ReAct Action Set, allowed review decisions, evidence and citation validation, approval state, or Tool Gateway behavior. For model-bearing workflow stages, the addendum may enter the Harness-normalized model request as structured user/context payload. For non-model governed stages, it may affect adjacent reviewed summaries, deterministic wording preferences, or trace-safe configuration summaries, but not retrieval, tool, memory, policy, validator, or response execution logic.

The Dashboard Workflow module uses the Dashboard Workflow Lens: a Workflow Template summary, read-only Workflow Relationship Map, Stage Inspector, and advanced YAML artifact. The map shows stage order, branch conditions, predecessor and successor relationships, inputs, outputs, and governed handoff points so users can understand how one stage's context affects adjacent execution. The inspector edits only descriptor-allowed Prompt and context fields for the selected stage and shows locked Harness boundaries. A Workflow Stage Context Preview may render a redacted, length-bounded sample of the effective context layers, but it must not call a model, execute a tool, write run trace, or become a second execution path.

Prompt configuration is Draft Agent state and must be validated before save, validation, and publication. Validation checks stage ids against the descriptor, fields against editable Prompt allowlists, context options against stage allowlists, length and item limits, secret-looking values, raw chain-of-thought requests, policy or approval bypass wording, direct tool execution wording, and runtime assembly paths that would replace Harness-owned control prompts. Agent Validation Runs include Workflow Stage Prompt Validation, and Agent Publication requires it to pass.

Run trace, Governance Receipt, and Dashboard Run Detail record only trace-safe summaries of applied stage Prompt configuration, such as stage id, configured field names, context option names, bounded counts, lengths, and whether redaction was applied. They do not store or render full stage Prompt text by default. Full Prompt content belongs to Draft or Published Agent Contract views, not ordinary run artifacts.

When an Agent is published, `workflow.stages[]` is frozen into the Published Agent Version's Agent Contract Bundle together with the selected Workflow Template Descriptor Version. Descriptor defaults may initialize Drafts or guide the UI, but production runs do not dynamically inherit latest descriptor Prompt text. Historical Published Agent Versions must remain explainable even if the backend template descriptor later evolves.
