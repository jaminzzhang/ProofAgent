# Dashboard Run Detail Readability Overhaul

Accepted.

The Dashboard Run Detail page currently presents a run as a stack of stage cards with badge counts and a raw JSONL trace, which forces an operator to reconstruct the run's shape, the ReAct loop length, and the Refusal point by reading trace events. We will add a Run Flow Diagram to the Workflow tab and a LLM Interaction Message View to the Validation Capture panel so the same data answers "which stages ran, how many reasoning cycles, where did it refuse, and what context was sent to the model" without parsing events or JSON.

The Run Flow Diagram renders Workflow Stage nodes left to right as a directed flow. Each node encodes three channels: border color for terminal outcome (green answered, red refused or failed, yellow weak or waiting, grey intermediate), fill and border style for visited state (solid fill when visited, dashed hollow grey for configured-only), and a self-loop badge for the ReAct Self-Loop Iteration Count. Configured-only stages are drawn rather than hidden so a refusal run visibly shows the answer stage that never executed. The diagram is hand-built SVG and CSS with no new graphing dependency, because the repository keeps frontend dependencies minimal and the node count per run is small enough that auto-layout is unnecessary.

The ReAct Self-Loop Iteration Count is the number of `reasoning_summary` trace events linked to a stage, not the number of `action_proposal` events. Refusal paths that skip reasoning render with count zero and no self-loop, and the refusal terminal node is visually distinct, so an absent loop reads as "did not deliberate, refused early" rather than a broken diagram. This is chosen over counting `action_proposal` events because a retried action that shares an `action_id` across iterations would over-count, and over falling back to `action_proposal` when reasoning is absent because that would erase the meaningful signal that a refusal run never engaged the loop.

The LLM Interaction Message View renders each Workflow Stage LLM Interaction Capture's `request_json.messages` array as a vertical sequence of role-headed cards (system, user, assistant) showing the original content verbatim with syntax highlighting, character count, and copy affordance, and renders the response as a highlighted JSON block with no field-level decomposition. We render content verbatim rather than decomposing JSON-as-prompt fields because run auditing requires the unchanged original string, any field decomposition is a rendering contract that rots as prompt templates evolve, and syntax highlighting already makes the structure legible. Failure diagnostics continue to use the existing human-readable diagnostic copy plus the raw JSON block.

This scope covers validation runs only. Production runs do not produce a `validation_capture_id` and therefore have no captured LLM request or response bodies, which is a Control Envelope governance boundary, not a Dashboard gap. The Capture tab remains absent on production runs, and we do not extend body capture to production runs in this work because that touches a sensitive retention class and requires its own decision rather than being smuggled through a readability change.
