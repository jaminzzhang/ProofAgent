# Local PageIndex Knowledge Source Ingestion

Proof Agent will add `local_pageindex` as a distinct Knowledge Provider for Dashboard-managed document ingestion into locally persisted PageIndex tree indexes. The existing `pageindex` provider remains the remote retrieval-endpoint integration. Dashboard uploads create reusable PageIndex-Backed Knowledge Sources with explicit PENDING, READY, or FAILED index state, and Agents may bind only READY sources through Agent Knowledge Bindings. Knowledge Sources own provider configuration; Agents bind sources without selecting providers. A source is a document collection: each stable Knowledge Document identity owns immutable revisions, and each revision is indexed independently through `QUEUED`, `PROCESSING`, `READY`, or `FAILED` state while Agent bindings resolve an immutable READY Knowledge Source Snapshot. Dashboard requires explicit new-document or replacement intent for a same-named upload; filename never causes implicit overwrite. Explicit replacement creates a new revision id. A repeated upload with the same content hash and compatible ingestion configuration reuses the existing index artifact as an idempotent operation. Failed revisions may be retried, replaced, or archived independently. A candidate snapshot includes READY revisions only, requires at least one READY revision before publication, and explicitly lists excluded processing, failed, and archived revisions in Dashboard. Until a replacement reaches READY, the candidate snapshot continues to use the prior READY revision. After replacement becomes READY, only the unpublished candidate snapshot changes. One failed revision does not silently disappear and does not permanently block publication of unrelated READY revisions. Failed replacement builds preserve the last READY snapshot. Upload, replacement, and removal prepare candidate snapshots; an explicit Knowledge Source Publication action is required before a candidate snapshot becomes available to Agent bindings. Draft Agents may preview a source's latest published snapshot, but each Published Agent Version captures the exact snapshot id used for validation and publication so later source changes cannot silently alter production behavior or break Agent version rollback. Ingestion work is represented by persisted jobs claimed by a separate local worker process through a file-backed recoverable queue boundary; production deployments may replace that queue without changing Knowledge Source semantics. Each PageIndex-Backed Knowledge Source owns a separate ingestion model configuration with credential environment-variable references only. Its Provider tab exposes common ingestion and inherited routing-model settings first, with timeout, retry count, worker concurrency, and routing-model override in advanced settings. A model-configuration test validates settings and referenced credential availability without rebuilding indexes. An artifact-affecting ingestion-configuration change marks the Source Draft as requiring reingestion; existing published snapshots remain usable, but a replacement candidate snapshot cannot publish until required revisions are reingested successfully. A stable ingestion-configuration fingerprint plus document content hash identifies compatible reusable index artifacts. Incremental reingestion queues only candidate-snapshot revisions that lack compatible artifacts. Routing-model changes, document selection budget changes, and routing-metadata edits do not rebuild indexes. Dashboard shows the required-rebuild count and exposes a `Reingest required documents` action; every required revision must be READY before replacement snapshot publication. The local worker defaults to per-Source concurrency 2 configurable from 1 through 8 and performs at most 2 automatic retries per revision with backoff. Model timeouts, transient rate limits, and temporary network failures are recoverable. Unsupported formats, scanned PDFs, missing configuration, and missing credentials fail immediately until operator action. Exhausted revisions enter FAILED with stable Dashboard error codes and short explanations and remain manually retryable. Worker restart resumes unfinished persisted tasks and checks for compatible artifacts before building to avoid duplicate completed work. Local PageIndex ingestion remains optional and cannot become a dependency of deterministic demos or default CI. V1 Dashboard operator intake accepts text-based PDF and Markdown only after server-side MIME and content-signature validation rather than trusting filename extension alone. One file is limited to 50 MB, one PDF to 500 pages, and one batch to 50 files while Source capacity remains 500 documents. Intake rejects zip archives, directory upload, nested attachments, encrypted PDFs, scanned PDFs, macro-bearing files, executable content, and unsupported formats. Files enter quarantine first under system-generated storage paths with sanitized display filenames. A validation failure reports a stable error code and creates neither revision nor ingestion job, so it cannot enter a candidate snapshot. Rejected quarantine files remain available for troubleshooting for 24 hours and are then removed automatically without promotion into long-term managed storage. Each validated revision retains its original PDF or Markdown separately from generated index artifacts for reingestion, citation verification, and audit-safe inspection. Operators with `knowledge_source.view` may download a managed original, and that download is audited. Archive and replacement retain originals while snapshots, Published Agent Versions, or audit-retention rules reference them; physical cleanup removes originals and index artifacts only after those checks pass. Local citations use stable internal URIs such as `knowledge://source/{source_id}/document/{document_id}/revision/{revision_id}#page=12`, or Markdown section anchors, without exposing storage paths. Dashboard Run Detail and retrieval preview resolve those URIs through permission-protected audited previews. Customer-facing citations project safe source name plus page or section and omit internal revision identifiers and storage paths. Source publication validation is bound to one Source Draft version and becomes stale after relevant configuration changes. Local PageIndex publication requires at least one READY revision, no pending required reingestion, compatible artifacts for every candidate-snapshot revision, successful ingestion and routing model tests, and an editable smoke query proving routing, retrieval, and citation resolution. Operator intake remains separate from customer chat intake. Document removal is archival: physical cleanup is allowed only for revisions and index artifacts unreferenced by retained snapshots and Published Agent Versions, preserving rollback. V1 supports up to 500 documents per Knowledge Source, so retrieval must route the query to a bounded set of eligible document revisions within the resolved snapshot before document-level PageIndex search. Routing filters operator-managed metadata and then uses an LLM selector over filenames plus editable ingestion-generated descriptions; its audit-safe selection summary is recorded in the governed retrieval plan trace. The source-specific routing model inherits the ingestion model configuration by default and may be overridden independently. The document selection budget defaults to 8 and is configurable from 1 through 20. Empty routing returns no evidence rather than silently widening search. Routing failure and any selected-document search failure fail closed instead of silently returning a partial result.

We chose separate providers because local ingestion owns uploaded-file storage, indexing state, and local workspace persistence, while remote PageIndex retrieval owns endpoint and credential configuration. Combining them behind one provider name would make the Agent Contract ambiguous and couple local design-time ingestion to remote runtime connectivity.
