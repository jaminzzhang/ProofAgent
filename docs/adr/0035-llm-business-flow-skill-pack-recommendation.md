# LLM Business Flow Skill Pack Recommendation With Control Plane Admission

Accepted.

Proof Agent will move Business Flow Skill Pack routing from post-Intent-Resolution string matching to an LLM-authored **Business Flow Skill Pack Recommendation** produced from the same Intent Resolution model response as the **Intent Resolution Contract**. The recommendation is an independent fact, not a field inside the Intent Resolution Contract, and it never grants execution authority. The Control Plane remains responsible for Business Flow Skill Pack Admission, including existence, readiness, authorization posture, confidence gates, ambiguity handling, trace, receipt, and evaluation semantics.

This is a corrective replacement for `react_enterprise_qa.v3`, not a new `react_enterprise_qa.v4` descriptor version. The current V3 Business Flow Skill Pack routing behavior is considered a defective implementation of the intended V3 semantics because it performs brittle post-intent string matching, converts missing matches into user clarification, and permits default-pack fallback semantics that can behave like a hidden catch-all. Future V3 executions should use the recommendation-and-admission semantics in this ADR. Existing run artifacts remain auditable historical facts and must not be recomputed or reinterpreted into the new semantics, but V3 should not preserve the defective routing behavior as a supported compatibility contract.

Intent Resolution may see only **Business Flow Skill Pack Routing-Safe Summary** values from the Published Business Flow Skill Pack Set: identity, label, description, intent patterns or taxonomy references, admission hints, and bounded capability reference counts or summaries. It must not see full pack YAML, stage Prompt addenda, policy details, validator details, tool schemas, or downstream stage guidance before admission. This keeps recommendation and post-admission context application separate.

The recommendation contract uses an explicit `recommendation_type` with V1 values `single_pack`, `no_pack`, and `ambiguous`. It carries top-level **Business Flow Route Confidence** and bounded route rationale. Candidate packs are represented only as structured `candidate_packs[]` entries, each with `pack_id`, candidate-level confidence, and bounded rationale, ordered by confidence descending. There is no parallel `recommended_pack_id`; `single_pack` requires exactly one candidate, `ambiguous` requires two or more, and `no_pack` requires none.

`no_pack` is a normal, explicit recommendation, not omitted data. When admitted as no-pack, the run becomes a **No Business Flow Skill Pack Run** and continues through the base Workflow Template without pack-specific context. Skills-disabled Agents do not prompt for or emit Business Flow Skill Pack Recommendation. Agents with `capabilities.skills.enabled: true` must publish a non-empty Published Business Flow Skill Pack Set; enabled skills with no published business flows fail publication validation.

Admission uses two confidence gates. The Agent skills admission configuration owns `route_min_confidence`; if Business Flow Route Confidence is below that threshold, the Control Plane records no-pack before considering route type or candidates. Each Business Flow Skill Pack owns its candidate `min_confidence`; if a `single_pack` candidate fails that pack-level threshold, the run proceeds as no-pack rather than weakly admitting the pack or asking the user to confirm a pack name.

V1 admits at most one **Primary Business Flow Skill Pack** per governed run. Composite requests are handled through a purpose-built composite pack, a No Business Flow Skill Pack Run, or a clarification asking the user to split or choose the intended task. V1 does not merge multiple packs, union capability references, combine stage addenda, or stack authorization contexts. If an `ambiguous` recommendation represents materially different candidate packs, especially a composite request with different capability boundaries, the Control Plane requests clarification or task split. The LLM may provide ambiguity and materiality hints, but the Control Plane makes the final materiality decision from Published Business Flow Skill Pack metadata; if materiality cannot be safely determined, it clarifies.

Malformed required recommendations fail closed as model output contract failures and must not be converted into no-pack. Non-repairable violations include missing required recommendation output when skills are enabled, unsupported recommendation type, invalid candidate shape, missing pack ids, non-numeric confidence, and out-of-range confidence. Candidate ordering mistakes are repairable: the Control Plane may reorder candidates by confidence descending and record Business Flow Recommendation Normalization.

Trace and projections should preserve the independent facts: `intent_resolution`, `business_flow_skill_pack_recommendation`, optional recommendation normalization, and deterministic `business_flow_skill_pack_admission`. Governance Receipt, Dashboard, and evaluation gates should distinguish normal no-pack routing, low route confidence no-pack, low candidate confidence no-pack, ambiguous clarification, task-split clarification, contract failure, and unauthorized or not-ready fail-closed outcomes.

We choose this over deterministic substring matching because routing-safe pack summaries are semantic business configuration and should participate in the same Intent Resolution prompt as the user request and conversation context. We choose Control Plane admission over LLM selection because pack admission can alter stage context and governed capability references. We reject automatic Default Business Flow Skill Pack fallback because it turns a broad pack into a hidden catch-all and can silently change business context. We reject multi-pack merge in V1 because merged stage addenda, capability references, policy posture, and authorization boundaries are not safely composable inside one run.
