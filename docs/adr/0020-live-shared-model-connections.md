# Live Shared Model Connections for Agent and Knowledge Configuration

Proof Agent will add a Dashboard Models Workspace for Shared Model Connections that Agents and Knowledge Sources reference directly for provider, model identifier, endpoint, and credential reference values. We chose live references rather than pinning model connection versions into Published Agent Versions because model access configuration is expected to change frequently and should be centrally updateable for all referencing Agents and Knowledge Sources.

This deliberately differs from Knowledge Source snapshot binding: model connection updates take effect immediately for existing references, while Model Connection Impact Review, configuration audit, Model Connection Resolution Records, and fail-closed resolution behavior make those updates observable and accountable. Model Usage Parameters remain owned by the referring Agent role or Knowledge Source, and custom model configuration remains available when a specific Agent role or Source should not use a Shared Model Connection.

Model Connection Impact Review is required for high-impact Shared Model Connection changes: provider, model identifier, base URL, credential reference, default timeout, archive, restore, and deletion. Display name, description, and tags are ordinary metadata edits that write configuration audit but do not require the pre-save impact confirmation.

Archived Shared Model Connections are hidden from new selectors and cannot be used for new Agent publication or new production-bound Knowledge Source publication, but existing Published Agents and existing Knowledge Source runtime use may continue resolving them with warnings until an operator switches the reference or restores the connection. Physical deletion remains allowed only when no Draft Agent, Published Agent, or Knowledge Source references remain.

Agent Validation Runs may execute against archived Shared Model Connections so operators can assess current behavior, but the validation result carries a publish-blocking warning and Agent Publication fails until the Draft Agent switches to an active connection, uses custom configuration, or the connection is restored. If the archived connection cannot resolve or call successfully, validation fails normally.

The Models Workspace list is an operational inventory. It shows display name, connection id, provider, model identifier, base URL host, credential reference, lifecycle state, Agent and Knowledge Source reference counts, last smoke-test status and time, and update time, with filters for provider, lifecycle, reference state, smoke-test status, and text search.

The Shared Model Connection detail workspace uses lightweight Overview, References, Test, and Audit tabs. Overview edits connection parameters and metadata; References shows Draft Agent, Published Agent, and Knowledge Source live references and high-impact change previews; Test runs local validation and manual smoke tests; Audit lists configuration changes with emphasis on high-impact fields.

Reference counts are configuration references, not historical usage counts. Draft Agent references scan Draft Agent contract YAML, Published Agent references scan Published Agent Version contract snapshots for live `connection_id` references, and Knowledge Source references scan Source-owned ingestion and routing model configuration. Run history is represented through Model Connection Resolution Records rather than reference counts.

Agent Custom Model Configuration and Knowledge Source Custom Model Configuration may be explicitly copied into a new Shared Model Connection through a "save as shared" flow. The flow pre-fills connection fields from the custom configuration, creates a separate reusable connection, and only switches the current Agent role or Knowledge Source reference when the operator confirms; it does not copy behavior-changing usage parameters except default timeout.

V1 does not provide Shared Model Connection import or export. Operators configure model connections directly in the target environment, keeping the scope focused on live configuration, reference impact review, validation, smoke testing, and audit rather than portable connection manifests.

Shared Model Connection identifiers are stable references. The Models Workspace generates `connection_id` values from display names by default, allows explicit operator-provided ids at creation time, restricts them to a safe identifier character set, and makes them immutable after creation; later renaming changes `display_name` only.

Creating or editing a Shared Model Connection may save an environment credential reference even when the current runtime environment does not define that variable. Model Connection Validation, Model Connection Smoke Test, Agent and Knowledge execution, and ingestion or routing jobs fail or warn explicitly when the credential reference cannot resolve in the runtime environment; smoke tests do not call remote providers when credentials are missing.

Physical deletion requires the connection to be archived, have zero Draft Agent, Published Agent Version, and Knowledge Source configuration references, and have no in-flight validation or smoke-test operation. Deletion writes root-level configuration audit before removing connection storage so the deletion fact survives removal.

V1 credential configuration is limited to environment-variable Model Credential References; Proof Agent does not store raw provider API keys or provide a Dashboard-managed secret vault. The first implementation targets Dashboard-managed Draft Agents and Knowledge Sources while preserving standalone Agent Package YAML and deterministic examples that define inline model configuration.

Agent and Knowledge configuration use an explicit model-source discriminator instead of inferring intent from missing fields. Dashboard-managed model call roles may set `model_source: shared` plus a `connection_id`, or `model_source: custom` plus provider, model identifier, endpoint, and credential reference fields; legacy standalone package YAML that defines inline `provider` and `name` remains supported for self-contained demos and packages.

Shared Model Connection records store `connection_id`, `display_name`, optional description and tags, provider, `model_identifier`, plaintext `base_url` when needed, environment credential reference, optional provider account-scope environment references, optional default `timeout_seconds`, lifecycle state, and timestamps. `display_name` names the reusable connection for operators, while `model_identifier` names the provider model; raw API keys are never stored in the connection record. If an Agent role or Knowledge Source configures its own `params.timeout_seconds`, that call-site usage parameter overrides the connection default. V1 does not store other usage defaults, such as temperature, maximum output tokens, retrieval `top_k`, reviewer controls, or ingestion and routing tuning, on the Shared Model Connection.

As part of this contract cleanup, Reviewer model usage settings move into the same `params` shape used by Answer and Planner model roles. Reviewer control-policy fields such as `fail_closed` remain on `review.subagent`, while model usage parameters such as timeout, maximum output tokens, and temperature live under `review.subagent.params`.
