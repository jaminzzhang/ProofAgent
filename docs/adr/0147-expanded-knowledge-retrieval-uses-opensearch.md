# Expanded Knowledge Retrieval Uses OpenSearch

Accepted.

[FRAME | HIGH] The production-directed Hybrid Index Provider uses self-hosted OpenSearch 3.x as the single Knowledge Retrieval Index Service for authorization-prefiltered lexical and dense-vector retrieval, deterministic metadata and effective-time filters, and Source-local reciprocal-rank fusion. A private external reranker receives only already-authorized candidates, and the existing Control Plane fusion remains a separate cross-Source stage. The provider translates typed Proof Agent requests into backend queries and returns normalized Candidate Evidence; OpenSearch scores never establish rule authority or Evidence Admission Score. Each physical index family is scoped by Source plus Knowledge Index Generation and serves multiple routine Source Publication Sequences through immutable Insurance Rule Unit Revisions and membership intervals. PostgreSQL and exact S3 Rule Unit Publication Manifests remain authoritative; immutable PostgreSQL-bound Knowledge Index Projection Attestation chain entries tie covered sequences to exact OpenSearch index UUID, generation, mapping, refresh checkpoint, manifest digests, counts, and parent proof. A historical binding accepts its publication-time attestation or a verified descendant covering the pinned sequence, so later same-generation publications can advance infrastructure state without changing historical corpus identity. Every physical generation remains rebuildable, and Local Index Provider remains a bounded migration path. We choose OpenSearch over scaling the current TreeIndex, a vector-only database, paid Elasticsearch features, or Vespa operational complexity for this corpus and team envelope.
