# Dashboard-Hosted Agent Configuration Workspace

Proof Agent will add an Agent Configuration Workspace inside the shared Dashboard Shell instead of building a detached builder app or turning the Dashboard API into an execution/configuration catch-all. We chose this because Agent owners need to move fluidly between monitoring a governed Agent and changing its configuration, while the trust boundaries still require separate API surfaces for observability, configuration, and execution.

The Dashboard Shell becomes Agent-centric: global run, approval, and handoff monitoring remains available, and each Agent detail view combines Monitor, Configure, Validate & Test, Versions, and Contract View. Configuration is served by an Agent Configuration API; Dashboard API remains read-only observability; Run Execution API and Customer Run API continue to execute only Published Agent Versions.

Agent configuration follows a Draft Agent lifecycle. The Agent Configuration Store owns drafts, version history, validation results, publication metadata, reusable Knowledge Sources and Tool Sources, and operation audit metadata. Agent Contract and Agent Package artifacts remain the reviewable execution language; existing Agent Packages can be imported into Draft Agents without overwriting the source package by default.

Publication promotes a validated Draft Agent into an immutable Published Agent Version after an Agent Validation Run. A Published Agent resolves to an Active Agent Version by default, and rollback changes that active pointer instead of mutating or deleting historical versions. Run artifacts must record the actual Published Agent Version used for execution.

Workflow editing is Workflow Template Node Configuration, not free-form runtime graph editing. The UI may expose node-based editing for planner, retrieval, review, tool gate, answer gate, memory, and response settings, but those settings compile back into registered Workflow Template and Agent Contract semantics.

Knowledge and tools use a reusable asset plus Agent binding model. Knowledge Sources and Tool Sources are reusable; an Agent Knowledge Binding Set and Agent Tool Bindings define per-Agent scope, retrieval strategy, approval behavior, and authorization constraints. An Agent may bind multiple Knowledge Sources for governed blended retrieval while each Knowledge Source owns its provider configuration. Memory remains Agent-specific Agent Memory Configuration in the first implementation stage because consent, lifecycle controls, and cross-Agent leakage risk make reusable memory assets a later platform concern.

Knowledge Source administration and Agent lifecycle administration preserve distinct capabilities even when the first local single-user experience grants them all by default. Knowledge Source actions distinguish `knowledge_source.view`, `knowledge_source.edit`, `knowledge_source.publish`, and `knowledge_source.archive`; Draft Agent binding changes require `agent.edit`, and Agent publication requires `agent.publish`. API checks, Dashboard action visibility, and configuration operation audit records keep these boundaries explicit for future RBAC.

Knowledge Source administration writes configuration operation audit records separately from retrieval runtime facts. The configuration history covers uploads, replacements, retries, document archives, Source publication, Source archive and restore, remote verification, binding changes, retrieval overrides, and explicit upgrades. Run detail surfaces read trace-safe runtime facts for resolved versions, routing, provider status, degraded retrieval, fusion, admission, citation, and context-budget truncation without exposing raw documents, secrets, or complete remote responses.

The existing Dashboard `/knowledge` route evolves into the global Knowledge Source Workspace for reusable Source administration. It replaces the current read-only cross-Agent knowledge configuration browser; Proof Agent does not add a parallel `/knowledge-sources` route. Each Source links to `/knowledge/:sourceId`, whose Overview, Documents, Versions, Provider, and Audit tabs centralize reusable-asset administration. Agent Knowledge modules remain focused on selecting multiple Sources, adjusting bounded binding settings, showing resolved versions, applying explicit upgrades, and linking to Source detail without embedding complete file management. Source publication and Agent publication remain separate actions.

The `/knowledge` list is an operational projection rather than a thin navigation page. Each row shows Source name, description, tags, provider type, active or archived lifecycle, local index or remote verification availability, current published snapshot or configuration version, local READY and total document counts or remote target index or namespace, referencing Agent count, and warning indicators for unpublished changes, failed ingestion, or stale remote verification. Operators may filter by name, tag, provider, lifecycle, and warning state.

Creating a Source from `/knowledge` uses a guided Source Draft wizard that branches immediately by intake type. Upload local documents creates a `local_pageindex` Source and collects ingestion-model configuration before PDF or Markdown upload. Connect remote knowledge selects `pageindex`, `http_json`, or another registered adapter and collects endpoint, environment-variable credential references, target index or namespace, and mappings as supported by the descriptor. Register existing local source preserves `local_markdown` and `local_vector` for development, migration, and deterministic demos. The wizard saves a Source Draft only; verification and explicit Source publication remain separate actions.

Publish Source opens a reviewed confirmation step rather than activating immediately. It shows Source Draft and prior published version, local document additions, replacements, archives, and READY count or remote adapter, target, pinned or mutable-external consistency mode, and latest verification time, plus smoke-query result and referencing Agent count. Publication requires a `change_note` stored in version history and configuration audit. The confirmation states that Source publication does not mutate existing Published Agent Versions; it only makes an upgrade available to Draft Agents.

The Source Versions tab lists every published snapshot or configuration version with publication time, actor, `change_note`, validation result, referencing Agent count, and version diff actions. Rollback does not mutate history or rewind a shared production pointer. It creates a new Source Draft from the selected historical version, then requires review, fresh Source validation, and explicit Source publication to produce a new version. The rollback flow never changes Published Agent Versions or Draft Agent bindings automatically; Agents adopt the newly published Source version only through explicit binding upgrade, Agent validation, and Agent publication.

Every Source supports audited secret-free manifest export with metadata, provider type, parameters, environment-variable references, version information, mappings, and routing configuration. A local PageIndex Source may additionally export an explicit offline bundle containing manifest, validated originals, and content hashes without credential values or cached indexes by default. Import always creates a Source Draft. Remote imports require fresh connection testing, validation, and publication. Local bundle files pass upload validation again and reingest according to the ingestion-configuration fingerprint; Proof Agent never trusts externally supplied index artifacts directly.

The Agent Configuration API exposes shared Source resources under `/api/config/knowledge-sources`. Collection operations list, create, and filter Sources. `/api/config/knowledge-sources/:sourceId` reads detail, updates a Source Draft, archives, and restores. Nested document resources cover upload, replace, retry, archive, metadata, and revision history. Nested version resources cover list, diff, and rollback-Draft creation. Source actions cover validate, publish, test connection, preview retrieval, export, and import. Agent Draft configuration stores `knowledge_bindings[]` plus Agent-level blended-retrieval settings only; provider parameters remain Source-owned. Published Agent Versions capture resolved Source snapshot or configuration versions and resolved binding settings.

Because no Agent deployment requires compatibility, the workspace and contract migrate directly. Dashboard editors, loader, examples, fixtures, and tests move together to `knowledge_bindings[]`; the new Agent contract rejects inline `knowledge.provider + params` rather than preserving a legacy editor or runtime path.

Implementation starts below the UI: first migrate contracts and loader, then add the Knowledge Source store and API, then local PageIndex ingestion, then multi-source retrieval runtime. The Dashboard `/knowledge` workspace, Source detail tabs, Agent Knowledge binding editor, and Run Detail projections build on those boundaries, followed by fixture and test migration.

For `local_pageindex`, the Documents tab is a paginated operating surface for up to 500 documents rather than a raw upload list. It supports batch PDF and Markdown upload; filters for name, tag, and state; and row-level visibility into document name, current READY revision, pending revision, state, update time, and current-published-snapshot membership. Single-document actions cover replace, failed-revision retry, routing-metadata edit, archive, and revision history. Bulk actions cover failed-revision retry, archive, and tag edit. A persistent candidate-snapshot summary shows included READY, processing, failed, and archived counts alongside the explicit Publish Source action.

For remote Sources, the Provider tab uses layered descriptor-driven configuration. The default form exposes adapter, endpoint, environment-variable credential references, index or namespace, timeout, and default `top_k`. An advanced section exposes supported protocol version, request mapping, response JSON Pointer mapping, and structured `source_ref` mapping. Typed adapters such as `pageindex` prefer focused forms; `http_json` exposes bounded mapping editors. Test connection validates connectivity, authentication, target existence, and normalization shape. Preview retrieval runs a bounded example query and shows normalized candidates, citations, provider-native scores, and revision observations. Neither action publishes a Source; saved Draft changes still require successful verification and explicit Source publication.

For `local_pageindex`, the Provider tab also uses layered configuration. The default form exposes ingestion model provider, model, environment-variable credential references, routing model inherited from ingestion by default, and document selection budget defaulting to 8. Advanced settings expose an explicit routing-model override, timeout, retry count, and worker concurrency. Documents and routing metadata remain in the Documents tab. Test model configuration validates settings and referenced credential availability without rebuilding indexes. Any ingestion-configuration change that could affect generated artifacts marks the Source Draft as requiring reingestion; existing published snapshots remain usable, but a replacement candidate snapshot cannot publish until required revisions are reingested successfully.

The first experience uses an Agent Creation Wizard for setup, then a module-based workspace with visual forms, Workflow Template Node Configuration, Validate & Test, Versions, and Contract View. Contract View is an advanced view over the same Draft Agent state, not a second source of truth. Natural-language descriptions may help users understand configuration, but executable policy remains structured Policy Rule Configuration.
